# สถาปัตยกรรม (Architecture & Philosophy)

Universal Inventory ถูกสร้างขึ้นภายใต้กฎเหล็ก 2 ข้อ:
1. **Engine Independence:** ระบบต้องทำงานได้โดยไม่ผูกติดกับโหนดของเกมหลัก (Node3D/Node2D)
2. **Separation of Concerns:** แกนกลาง (Core) จะต้องไม่รู้จัก UI และ UI มีหน้าที่แค่ "แสดงผล" สิ่งที่ Core บอก

## โครงสร้าง 3 ชั้น (3-Tier Architecture)

### 1. Core Layer (แกนกลาง)
ประกอบไปด้วยสคริปต์ที่ดูแลเรื่องข้อมูลล้วนๆ (Pure Data):
- `InventoryComponent`: ควบคุมสล็อตกระเป๋า ขนาดของกระเป๋า และตรรกะการจัดวางแบบ Grid
- `EquipmentComponent`: จัดการช่องสวมใส่อุปกรณ์ ควบคุมเงื่อนไขประเภทของสวมใส่
- **ความพิเศษ:** ชั้นนี้ไม่แคร์ว่าเกมคุณเป็น 2D หรือ 3D มันสนใจแค่ตัวเลขและ Logic

### 2. API / Service Layer (ชั้นบริการ)
ทำหน้าที่เป็น "ผู้จัดการ" รับคำสั่งจากระบบอื่นแล้วไปสั่ง Core อีกที:
- `InventoryAPI`: ฟังก์ชันสำเร็จรูป เช่น `sort_inventory`, `transfer_item`, `consume_item`
- `EquipmentAPI`: สำหรับสั่งใส่/ถอดอุปกรณ์

### 3. UI / View Layer (ชั้นแสดงผล)
เป็นส่วนที่นำ Core Data มาวาดขึ้นหน้าจอ:
- `InventoryUI` และ `EquipmentUI`
- **ความพิเศษ:** UI ของเรา "โง่" (Dumb UI) มันไม่คำนวณ Logic เอง มันแค่รับ Signal จาก Core มาวาดภาพ และส่ง Request (ผ่าน API) เวลามีคนคลิก

## Data Model (ข้อมูลไอเทม)
เราใช้ Resource เพื่อความเบาและรวดเร็ว
- **`ItemData`**: ข้อมูลพื้นฐาน (ชื่อ, รูป, ขนาด Grid)
- **`ItemModule`**: ชิ้นส่วนขยายความสามารถ (เช่น ทำให้เป็นของกินได้, ทำให้ตีบวกได้) 
- **`runtime_data`**: เป็น Dictionary เล็กๆ ที่แนบไปกับสล็อต เพื่อเก็บสเตตัสที่เปลี่ยนไปตามเวลา (เช่น ความทนทานที่เหลืออยู่, เวลาเน่าเสีย)
